# KeelWare Contract: SKR2 F429

## Goal

This document defines the first concrete `KeelWare` board contract for TackleBox.

The purpose of the contract is to make the board-specific flash-resident boundary explicit so higher-level `Boatswain`, `Semaphorio`, and `SpyGlass` behavior can evolve without rewriting board glue.

## Board Identity

- contract id: `keelware.stm32.skr2-f429.v1`
- board vendor: `BigTreeTech`
- board name: `SKR 2`
- mcu: `STM32F429VGT6`
- mcu family: `STM32F4`
- cpu: `Cortex-M4`
- framework: `stm32cube`
- core clock target: `168000000 Hz`

## Flash Layout Contract

- bootloader base: `0x08000000`
- bootloader size: `0x00008000`
- application base: `0x08008000`
- flash end: `0x08100000`
- default vector table offset for the primary app image: `0x8000`

Supported app-layout variants currently exercised in the repo:

- primary app image at `0x08008000`
- alternate app image at `0x08010000`
- full-flash app image at `0x08000000`

## Transport Contract

### SWD / recovery transport

- upload/debug transport: `stlink`
- debug target: `stm32f4x`
- probe class: `ST-Link V2` validated

### USB runtime transport

- application transport: USB CDC ACM
- application VID/PID: `0483:5740`
- bootloader transport: USB CDC ACM
- bootloader VID/PID: `0483:5741`

### Storage transport

- removable config storage: microSD via SDIO + FatFs
- boot-time config file path: `0:device.cfg`

## Boot Contract

- BootAnchor handoff request is stored in `RTC->BKP0R`
- application-boot request magic: `0xB00720A5`
- bootloader consumes the request before normal init and jumps early to the application if present
- application can request return to bootloader or application through the shared boot-mode contract

## Required KeelWare Services

The SKR 2 KeelWare implementation must provide these services upward:

1. clock and startup initialization
2. GPIO role binding and remapping support
3. USB CDC transport readiness and buffering
4. persistent config storage in internal flash
5. microSD config-source access
6. TMC transport wiring and verification path
7. boot-mode coordination with the BootAnchor
8. safe driver-enable and motion pulse plumbing for the reference axis path

## Current Runtime-Mappable Roles

The current SKR 2 implementation exposes these runtime-configurable pin roles:

- `x_uart`
- `x_dir`
- `x_step`
- `x_enable`
- `x_stop`
- `diag0`
- `diag2`
- `ps_on`
- `safe_power`
- `led`

## Current Runtime Configuration Surface

The current SKR 2 KeelWare contract accepts or persists these configuration categories:

- pin-role mappings
- logic/polarity flags
- motion timing
- boot host-wait timing
- TMC current and stall settings
- simulated load threshold

## Failure Rules

- safety-critical stop behavior must continue functioning without a host session
- missing host connection must not block a safe boot
- invalid or unavailable microSD config must fall back cleanly to flash or defaults
- TMC verification failures may block motion or enable paths, but they must not corrupt boot or recovery behavior
- bootloader/app transitions must tolerate disconnect/re-enumeration as the normal case

## Validation Requirements

Changes to this contract should be validated with:

1. `test/run_ci_checks.ps1`
2. `act -j validate-linux`
3. `test/run_bootloader_serial_tests.ps1`
4. `test/run_live_serial_tests.ps1`
5. `test/run_stlink_validation.ps1` when SWD or boot/recovery behavior changes

## Non-Goals

- defining every board pin number in this contract document
- locking the symbol names of the current implementation files
- claiming that `Boatswain` is already independently updatable in the final intended form

This contract defines the boundary that future refactors should preserve while the codebase is renamed and split.
